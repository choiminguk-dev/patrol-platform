# 주소판 OCR 개선 기록 (analyze-batch)

> **파일**: `src/app/api/analyze-batch/route.ts`
> **기간**: 2026-04-06 (1차~4차, 같은 날 반복 개선)
> **대상 데이터**: 환경순찰 44장 (상습지역 순찰, 04-03 / 04-05)
> **최종 상태**: 10개 주소 중 10개 정확 인식 (100%)

---

## 문제 정의

공무관이 순찰 중 촬영한 사진을 일괄 업로드하면 AI(Claude Haiku)가:
1. 사진 속 파란색 주소판의 도로명+건물번호를 OCR
2. 작업사진(1~5장) + 주소판(1장) 패턴으로 그룹핑
3. 각 그룹에 정확한 주소를 배정

**실제 발생한 오류 유형:**

| 유형 | 예시 | 원인 |
|------|------|------|
| 사진 미분석 | 31~44번 사진 무시 | 30장 하드코딩 제한 |
| 도로명 오독 | 두텁바위로47길 → 후암로34가길 | 이전 청크 이미지 프라이밍 |
| 건물번호 분해 | 두텁바위로 155 → 두텁바위로55길 2 | 3자리 번호를 길+번호로 분해 |
| 숫자 혼동 | 후암로 59 → 후암로 57 | 5/7/9 OCR 혼동 |
| 도로명 누락 | 두텁바위로35길 2 → 두텁바위로 2 | 작은 글자(35길) 미인식 |
| 그룹 합침 | 2개 그룹(8장)으로 합침 | 주소판 2개를 1개로 인식 |
| 그룹 분리 | 주소판만 1장 별도 그룹 | 작업사진과 주소판 분리 |
| 방향 오류 | 주소판 뒤 사진을 같은 그룹에 | 그룹핑 방향 혼동 |

## 개선 이력

### 1차: 30장 제한 제거 (15장 청크)

**문제**: `Math.min(photoUrls.length, 30)` → 44장 중 14장 미분석
**수정**:
- 15장씩 청크로 분할하여 순차 API 호출
- 각 청크 결과를 병합
- 프론트엔드 "누락 보정" 로직은 유지 (안전장치)

**결과**: 7건 → 12건으로 분리 (10건이 올바름). 31~44번 사진의 주소판 3개 인식 성공.

### 2차: 도로명 검증 + 청크 겹침 + 후처리 병합

**문제**: "두텁바위로 155" → "후암로57길 30-16" 완전 오독
**수정**:
- 후암동 도로명 11개 목록을 시스템 프롬프트에 추가
- 청크 간 3장 이미지 겹침(overlap) 추가
- 1장짜리 그룹(주소판만) → 앞 그룹과 자동 병합
- OCR 프롬프트: "파란 표지판에서만 읽기", "추측 금지"

**결과**: 오독 감소. 그러나 겹침이 역효과 — 이전 청크의 "후암로34가길 1" 사진이 다음 청크에서 "후암로34가길"을 프라이밍.

### 3차: 겹침 제거 + 도로명 후처리 교정

**문제**: 이미지 겹침(overlap)이 다음 청크의 OCR을 오염
**수정**:
- 이미지 겹침 완전 제거
- 대신 이전 청크에서 발견된 주소를 **텍스트로만** 참고 전달
- `correctAddress()` 함수 추가:
  - 우암로→후암로, 두턴바위로→두텁바위로 자동 보정
  - 유효하지 않은 두텁바위로XX길 → 가장 가까운 알려진 도로명 교정
- OCR 프롬프트:
  - "건물번호 155를 55길+2로 분해하지 마세요"
  - "Duteobawi-ro 영문 참고"
  - "7장 이상 그룹은 주소판 누락 재확인"

**결과**: "두텁바위로 155" 정확 인식. "후암로34가길" 프라이밍 제거됨.

### 4차: 그룹핑 방향 + 동일주소 병합 + 퍼지매칭 (최종)

**문제**:
- "두텁바위로 2" (35길 누락)
- 주소판 뒤 사진이 같은 그룹에 포함됨
- 동일 주소가 2개 항목으로 분리됨

**수정**:
- 프롬프트: **"주소판 뒤 사진은 다음 그룹"** 명시 (방향성 강화)
- 프롬프트: "두텁바위로 건물번호 <50이면 XX길 재확인"
- 후처리: 동일 주소 연속 항목 자동 병합
- `correctAddress()` 강화:
  - 유사 도로명 퍼지매칭 (문자 일치도 기반)
  - 후암로XX가길 패턴 매칭
  - 두텁바위로XX길 → 가장 가까운 숫자의 알려진 도로명 교정

**결과**: 44장 → 10개 그룹, 10개 주소 모두 정확. 완벽 인식.

## 최종 시스템 구성

### 분석 파이프라인

```
44장 업로드
  ↓
[청크 분할] 15장씩 (1-15, 16-30, 31-44)
  ↓
[Claude Haiku OCR] × 3회
  시스템 프롬프트: 도로명 목록 + OCR 주의사항 + 촬영패턴
  이전 청크 주소를 텍스트로만 참고 전달
  ↓
[결과 병합] 3개 청크 → 전체 그룹 목록
  ↓
[후처리 1: 도로명 교정] correctAddress()
  우암로→후암로, 두턴바위로→두텁바위로
  유효하지 않은 도로명 → 퍼지매칭
  ↓
[후처리 2: 1장 그룹 병합]
  주소판만 1장 그룹 → 앞 그룹에 병합
  ↓
[후처리 3: 동일주소 병합]
  연속된 같은 주소 항목 → 하나로 합침
  ↓
[프론트엔드 누락 보정]
  AI가 빠뜨린 사진 → 가장 가까운 그룹에 추가
  ↓
[사용자 리뷰] 그룹별 주소/카테고리/메모 수정 가능
  ↓
[일괄 제출] DB 저장
```

### OCR 주의사항 (프롬프트에 포함)

| 규칙 | 이유 |
|------|------|
| 건물번호 3자리는 그대로 (155 ≠ 55길+2) | Haiku가 숫자를 도로명으로 분해 |
| "두텁바위로" ≠ "후암로" (Duteobawi-ro 참고) | 한글 글자 혼동 |
| 도로명 숫자(1길/35길/47길/55길) 정확히 구분 | 작은 글자 미인식 |
| 두텁바위로 건물번호 <50이면 XX길 재확인 | 본도로 vs 지선 구분 |
| 주소판 뒤 사진은 다음 그룹 | 그룹핑 방향 오류 방지 |
| 7장 이상 그룹은 주소판 누락 재확인 | 과도한 그룹 합침 방지 |
| 파란 표지판에서만 읽기, 추측 금지 | 할루시네이션 방지 |

### correctAddress() 교정 로직

```
입력: "두텁바위로33길 2"
  ↓ KNOWN_STREETS에 "두텁바위로33길" 없음
  ↓ 두텁바위로XX길 후보: 1길(diff=32), 35길(diff=2), 47길(diff=14), 55길(diff=22)
  ↓ 가장 가까운 35길 선택
출력: "두텁바위로35길 2"
```

```
입력: "우암로34가길 9-1"
  ↓ 우암로→후암로 보정
출력: "후암로34가길 9-1"
```


### OCR 오류 발생 시 점검 순서
1. 해당 사진의 주소판 직접 확인
2. `KNOWN_STREETS`에 도로명이 있는지 확인
3. 없으면 추가, 있으면 프롬프트 OCR 주의사항에 힌트 추가
4. `correctAddress()`에 특수 교정 규칙 추가 가능
5. 서버 재빌드+재시작
